Create `code/webapp/src/components/DingTalkQRView.tsx`.

The component manages these states:
- `phase: 'loading' | 'iframe' | 'qrcode' | 'error' | 'success'` -- what the view shows
- `authUrl: string` -- the OAuth2 auth URL from backend
- `authState: string` -- the state parameter for polling
- `errorMessage: string | null` -- error text to display
- `refreshKey: number` -- increment to trigger re-fetch of auth URL

Component lifecycle:
1. On mount (and when refreshKey changes):
   a. Call `getDingTalkLoginUrl(serverUrl, 'poll')` -- UXP mode uses poll per D-20
   b. Store authUrl and authState in state
   c. Set phase to 'iframe' (try iframe first per D-04)
   d. Create an AbortController for polling

2. iframe attempt:
   a. Render an iframe with `src={authUrl}`, `width="100%"`, `height="320px"`, `frameBorder="0"`, `sandbox="allow-scripts allow-same-origin allow-forms allow-popups"` per D-05 (loaded directly by WebView, not Bridge)
   b. Set a 5-second timeout: if iframe `onLoad` doesn't fire, switch to 'qrcode' phase
   c. If iframe `onError` fires, switch to 'qrcode' phase
   d. If iframe `onLoad` fires successfully, stay on 'iframe' phase

3. QR code fallback (or primary if iframe was skipped):
   a. Render `<QRCodeSVG value={authUrl} size={200} level="M" />` from qrcode.react
   b. Display text "请使用钉钉扫描二维码登录" below the QR code
   c. The QR code is a static image of the auth_url -- user scans with phone camera

4. Polling (runs regardless of iframe vs QR display):
   a. Call `pollDingTalkAuth(serverUrl, authState, { signal: abortController.signal })`
   b. On success: call `loginWithDingTalk(serverUrl, result.data)`, then `onSuccess()`
   c. On error with message 'POLL_TIMEOUT': show error "登录超时，请重试" per D-28
   d. On error with message containing 'NetworkError' or 'fetch' or 'Failed to fetch': show "网络连接失败" per D-27
   e. On other errors: show error message from the Error object (DingTalk service errors per D-25, cancelled per D-26)

5. Auto-refresh (per D-10):
   a. Set a 3-minute timer (180000ms) after getting auth_url
   b. When timer fires: abort current polling, increment refreshKey to trigger new auth_url fetch
   c. Clear timer on unmount

6. Retry button:
   a. In error state, show a button "重新获取" that increments refreshKey
   b. This triggers a fresh getDingTalkLoginUrl + new polling cycle

7. Cleanup on unmount:
   a. Abort the AbortController (cancels pollDingTalkAuth)
   b. Clear the auto-refresh timer

The component should use `useEffect` with dependencies `[serverUrl, refreshKey]` for the auth URL fetch + polling cycle, and return a cleanup function that aborts and clears timers.

IMPORTANT: Per the CRITICAL RESEARCH FINDING -- the auth_url returned by `getDingTalkLoginUrl` uses the NEW OAuth2 API (`login.dingtalk.com/oauth2/auth`). Do NOT embed the old iframe page `login.dingtalk.com/login/qrcode.htm`. The backend generates the correct URL. The iframe attempts to load this NEW OAuth2 auth URL. If blocked, the QR code renders the same URL as a scannable image.

**DEVIATION FROM D-01:** CONTEXT.md D-01 specifies the old sns iframe URL (`login.dingtalk.com/login/qrcode.htm`), but this URL is incompatible with the backend's NEW OAuth2 API (returns `loginTmpCode` instead of `authCode`). Per D-04 risk mitigation path and RESEARCH.md Critical Finding, this plan uses the NEW OAuth2 auth URL (`login.dingtalk.com/oauth2/auth`) generated by the backend's `generate_login_url()`. This is technically correct and required for the flow to work. D-01's intent (display DingTalk QR in plugin) is preserved; only the URL mechanism is corrected.

Error handling per D-29: All errors show INSIDE the QR view with a retry button. Never switch back to password form automatically.

| T-07-07 | Tampering | browser redirect mode | mitigate | Use auth_url returned by backend directly, not user-constructed; redirect goes to backend callback which validates state |
