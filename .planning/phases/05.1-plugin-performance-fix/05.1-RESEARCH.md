# Phase 05.1: plugin-performance-fix - Research

**Researched:** 2026-04-16
**Domain:** UXP plugin performance optimization (keyboard focus, async base64, executeAsModal scope)
**Confidence:** HIGH

## Summary

Phase 05.1 fixes three diagnosed root causes in the Photoshop ComfyUI plugin: (1) keyboard shortcut failures caused by UXP WebView stealing focus from Photoshop on Windows, (2) UI freezing from synchronous base64 conversion and overly broad `executeAsModal` scope, and (3) slow uploads from redundant file I/O. The diagnosis in `.planning/debug/plugin-performance-shortcut-issues.md` is thorough and precise, with exact line numbers for all affected code.

The keyboard focus issue is a **confirmed Adobe UXP platform bug on Windows** -- clicking inside a WebView causes all PS shortcuts to stop working, even after clicking outside the panel. The workaround (D-01/D-02) is to intercept keystrokes in the webapp and forward them to Photoshop via the existing Bridge `postMessage` protocol, where `batchPlay` can execute equivalent menu commands. This is the only viable approach until Adobe fixes the underlying focus management.

The base64 conversion issue (D-03) affects three code locations in `main.js` and uses `String.fromCharCode.apply` in 32KB chunks synchronously. The fix is to add `await` yields between chunks. The `executeAsModal` scope reduction (D-04) is straightforward -- only the `batchPlay` calls need modal state; file I/O and string conversion can run outside.

**Primary recommendation:** Implement the three root causes as separate, independently-testable tasks. Start with the async base64 fix (biggest user-visible impact), then executeAsModal scope reduction, then the keyboard passthrough (most complex, requires Bridge message additions).

### Pattern 1: Keyboard Shortcut Passthrough (D-01/D-02)

**What:** Intercept keydown events in the webapp when the target is not an input/textarea, and forward recognized PS shortcuts to the plugin via Bridge.

**How it works:**
```
User presses key -> webapp keydown handler
  -> Check: is target an input/textarea? If yes, ignore
  -> Check: is key in PS shortcut whitelist? If no, ignore
  -> Send Bridge message: { action: 'ps.executeShortcut', payload: { key, ctrl, shift, alt } }
  -> main.js handler receives -> calls batchPlay with appropriate Photoshop action
```

**Shortcut-to-batchPlay mapping example:**
```javascript
// In main.js, new handler 'ps.executeShortcut'
const SHORTCUT_ACTIONS = {
  'Delete': [{ _obj: 'delete', _target: [{ _ref: 'layer', _enum: 'ordinal', _value: 'targetEnum' }] }],
  'Ctrl+Z': [{ _obj: 'undo' }],
  'Ctrl+S': [{ _obj: 'save' }],
  // ... etc
};
```

**Input field detection:**
```javascript
// In webapp keydown handler
const target = event.target;
const isInputField = target.tagName === 'INPUT' || target.tagName === 'TEXTAREA' || target.tagName === 'SELECT' || target.isContentEditable;
if (isInputField) return; // Don't intercept
```

### Pattern 2: Async Chunked Base64 Conversion (D-03)

**What:** Replace the synchronous `String.fromCharCode.apply` loop with an async version that yields to the event loop.

**Current code (main.js lines 345-352, 515-520, 837-842):**
```javascript
// SYNCHRONOUS - blocks entire thread for large images
let binary = '';
const bytes = new Uint8Array(fileData);
const chunkSize = 0x8000; // 32KB
for (let i = 0; i < bytes.length; i += chunkSize) {
  binary += String.fromCharCode.apply(null, bytes.subarray(i, i + chunkSize));
}
const base64 = btoa(binary);
```

**Replacement pattern:**
```javascript
async function arrayBufferToBase64(buffer) {
  const bytes = new Uint8Array(buffer);
  const chunkSize = 0x8000; // 32KB chunks
  let binary = '';
  for (let i = 0; i < bytes.length; i += chunkSize) {
    binary += String.fromCharCode.apply(null, bytes.subarray(i, i + chunkSize));
    // Yield to event loop every chunk to prevent UI freeze
    if (i + chunkSize < bytes.length) {
      await new Promise(resolve => setTimeout(resolve, 0));
    }
  }
  return btoa(binary);
}
```

**Why this works:**
- `setTimeout(resolve, 0)` yields control back to the event loop between chunks
- Other pending tasks (UI updates, user input) can execute during yields
- For a 16MB image: ~512 chunks of 32KB, each yielding -> total overhead is minimal but PS remains responsive
- The function remains `async` so callers already use `await` [VERIFIED: all three call sites are in async functions]

**Alternative considered:** Using native `Uint8Array.toBase64()` (TC39 proposal, now standardized). However, UXP runs an embedded Chromium that may not support this API yet. The manual chunked approach is safer and more portable. [ASSUMED -- UXP Chromium version unknown]

**Chunk size recommendation:** Keep 32KB (0x8000). This is already proven in the codebase. Larger chunks reduce yields but increase per-chunk blocking. For 8MB images, 32KB gives ~256 yields which is a good balance.

### Pattern 3: Reduced executeAsModal Scope (D-04)

**What:** Move non-batchPlay operations outside the `executeAsModal` wrapper.

**Current structure (e.g., exportActiveLayerPng, line 377):**
```javascript
return core.executeAsModal(async () => {
  // ALL of this is in modal state:
  // - Document duplication
  // - Layer visibility toggling
  // - Bounds checking
  // - Cropping
  // - Resizing
  // - PNG saving
  // - Base64 conversion  <-- especially expensive
  // - Visibility restoration
}, { commandName: 'Export Active Layer PNG' });
```

**Target structure:**
```javascript
// Phase 1: Non-modal - layer visibility setup (uses DOM API, may not need modal)
// Some DOM operations may still require modal state -- needs testing
const visibilityState = saveVisibilityState(activeDoc, activeLayer);

try {
  // Phase 2: Modal - only PS-mutating operations
  const exportResult = await core.executeAsModal(async () => {
    // Duplicate, crop, resize, save PNG -- all batchPlay/DOM mutations
    const duplicatedDoc = await activeDoc.duplicate();
    try {
      // bounds, crop, resize, saveAs.png
      return { tempPath: ... };
    } finally {
      await duplicatedDoc.closeWithoutSaving();
    }
  }, { commandName: 'Export Active Layer PNG' });

  // Phase 3: Non-modal - file I/O and base64 conversion
  const fileData = await exportedFile.read({ format: formats.binary });
  const base64 = await arrayBufferToBase64(fileData); // async, yields
  return { base64, ... };
} finally {
  // Phase 4: Non-modal - restore visibility
  restoreVisibilityState(activeDoc, visibilityState);
}
```

**Critical caveat:** Some Photoshop DOM operations (like `layer.visible = ...`, `activeDoc.duplicate()`, `doc.crop()`) may also require modal state. [VERIFIED: Adobe docs state "a modal state is required when a plugin wants to make modifications to the Photoshop state"](https://developer.adobe.com/photoshop/uxp/2022/ps_reference/media/executeasmodal/). This means layer visibility changes and document duplication DO need modal state. The practical win is moving the base64 conversion and file reading OUTSIDE modal scope.

**Realistic scope reduction:**
- Keep document manipulation inside executeAsModal
- Move `exportedFile.read()` + `arrayBufferToBase64()` OUTSIDE -- this is the biggest win since base64 conversion is the slowest part
- Move temp file cleanup to a finally block outside modal

### Anti-Patterns to Avoid
- **Forwarding all keystrokes:** Would break webapp input fields. Must use whitelist + input-target detection.
- **Running base64 conversion inside executeAsModal:** Blocks Photoshop UI for the entire conversion duration.
- **Using `requestAnimationFrame` for yields:** Not available in UXP plugin context. Use `setTimeout(resolve, 0)`.
- **Trying to handle Space/hold keys in v1:** Complex state tracking, defer to future iteration.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Shortcut-to-action mapping | Custom menu action IDs for every shortcut | batchPlay with well-known action descriptor objects | PS action descriptors are documented; custom approaches break across versions |
| Base64 encoding | Entirely new encoding algorithm | `btoa()` after chunked binary string building | `btoa()` is native and fast; only the string building needs async |
| Temp file management | Complex temp folder tracking system | Simple `finally { folder.delete() }` after export | UXP temp folders are isolated; no need for reference counting |
| File size without reading content | Custom binary parsing | UXP `entry.nativePath` metadata or approximate size from file properties | The current approach reads every file entirely -- even just skipping this is a win |

## Common Pitfalls

### Pitfall 1: executeAsModal Scope Reduction May Not Work for DOM Operations
**What goes wrong:** Moving `layer.visible = false` or `doc.duplicate()` outside executeAsModal throws "Photoshop is not in modal state" errors.
**Why it happens:** Adobe requires modal state for any Photoshop state mutation, not just batchPlay.
**How to avoid:** Test each operation individually. Keep all DOM mutations inside executeAsModal. Only move file I/O and string processing outside.
**Warning signs:** Errors like "The command is not available" or "Not in modal state" after refactoring.

### Pitfall 2: Keyboard Forwarding Breaks Input Fields
**What goes wrong:** PS shortcuts fire while user is typing in a prompt textarea.
**Why it happens:** keydown handler doesn't check if event.target is an input field before forwarding.
**How to avoid:** Always check `event.target.tagName` before forwarding. Test with text inputs, textareas, and contentEditable elements.
**Warning signs:** User reports typing "z" in a text field triggers undo.

### Pitfall 3: Async Base64 Race Condition
**What goes wrong:** Bridge response arrives before base64 conversion completes.
**Why it happens:** The async conversion is non-blocking, so the response handler might try to use incomplete data.
**How to avoid:** Ensure the entire async conversion completes before sending the Bridge response. The current code structure (await the conversion, then return) is correct -- just make sure no early returns are introduced.
**Warning signs:** Truncated base64 strings or corrupted images.

### Pitfall 4: Temp File Cleanup Fails During Modal State
**What goes wrong:** Trying to delete temp folders inside executeAsModal throws errors.
**Why it happens:** Some file system operations may be restricted during modal state.
**How to avoid:** Move temp folder cleanup to a finally block OUTSIDE executeAsModal scope.
**Warning signs:** Temp folders accumulating despite cleanup code.

### Pitfall 5: React.memo Without Proper Equality Checks
**What goes wrong:** React.memo with default shallow comparison still re-renders because parent passes new object/array references each render.
**Why it happens:** Default shallow comparison checks reference equality, not value equality. Objects like `style={{}}` create new references each render.
**How to avoid:** Use custom comparison functions where needed, or ensure parent components don't create inline objects/arrays in JSX. For the outputImages.map items, use stable keys and avoid inline arrow functions.
**Warning signs:** React DevTools profiler shows re-renders despite React.memo.

### Shortcut Execution Handler (main.js side -- new Bridge handler)
```javascript
// Source: derived from PS batchPlay action descriptor documentation
'ps.executeShortcut': async (payload) => {
  const { key, ctrl, shift, alt } = payload;

  // Map key combinations to batchPlay actions
  const shortcutKey = [
    ctrl && 'Ctrl',
    shift && 'Shift',
    alt && 'Alt',
    key
  ].filter(Boolean).join('+');

  const actionMap = {
    'Delete': [{ _obj: 'delete', _target: [{ _ref: 'layer', _enum: 'ordinal', _value: 'targetEnum' }] }],
    'Ctrl+KeyZ': [{ _obj: 'undo' }],
    'Ctrl+Shift+KeyZ': [{ _obj: 'redo' }],
    'Ctrl+KeyS': [{ _obj: 'save' }],
    'Ctrl+KeyC': [{ _obj: 'copy' }],
    'Ctrl+KeyV': [{ _obj: 'paste' }],
    'Ctrl+KeyA': [{ _obj: 'selectAll' }],
    'BracketLeft': [{ _obj: 'select', _target: [{ _ref: 'brush', _property: 'size' }], to: { _obj: 'size', width: { _unit: 'pixelsUnit', _value: -10 } } }],
    // ... more mappings
  };

  const actions = actionMap[shortcutKey];
  if (!actions) return { executed: false };

  return core.executeAsModal(async () => {
    await action.batchPlay(actions, { synchronousExecution: true });
    return { executed: true };
  }, { commandName: `Shortcut: ${shortcutKey}` });
},
```

## Open Questions

1. **Which Photoshop DOM operations require modal state?**
   - What we know: batchPlay definitely requires it. Document duplication and layer visibility changes likely require it.
   - What's unclear: Whether `exportedFile.read()` and `folder.delete()` work outside modal state.
   - Recommendation: Test incrementally -- first move only base64 conversion outside, verify it works, then move more.

2. **Does UXP support `setTimeout` in plugin context?**
   - What we know: UXP uses embedded Chromium with standard JS APIs. `setTimeout` is listed as available.
   - What's unclear: Whether `setTimeout(fn, 0)` actually yields to the event loop in UXP's runtime.
   - Recommendation: Test with a simple `await new Promise(r => setTimeout(r, 0))` in main.js and verify the UI remains responsive.

3. **Bracket key action descriptors for brush size?**
   - What we know: Bracket keys are standard PS shortcuts for brush resize.
   - What's unclear: The exact batchPlay descriptor for brush size adjustment.
   - Recommendation: Use Alchemist plugin to capture the correct action descriptor, or defer bracket keys to a follow-up.
