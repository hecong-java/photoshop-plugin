# Phase 05.1: plugin-performance-fix - Context

**Gathered:** 2026-04-16
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix critical performance issues in the Photoshop ComfyUI plugin that cause: keyboard shortcut failures, UI freezing during image export/import operations, and slow uploads. This is a performance optimization phase — no new features, only fixing diagnosed root causes and easy-win secondary issues.

Root causes diagnosed in `.planning/debug/plugin-performance-shortcut-issues.md`.

</domain>

<decisions>
## Implementation Decisions

### Shortcut / Focus Fix
- **D-01:** Add a keydown passthrough handler in the webapp that detects non-input PS keystrokes and forwards them to Photoshop via Bridge — avoids focus stealing by PS native shortcuts
- **D-02:** Forward only a whitelist of known PS shortcuts (Delete, Ctrl+Z, Ctrl+S, Ctrl+C, Ctrl+V, Ctrl+A, arrow keys, Space, bracket keys, etc.) — not all keystrokes — to avoid interfering with webapp input fields

### Thread Blocking Fix
- **D-03:** Replace synchronous `String.fromCharCode.apply` base64 conversion with async chunked processing that yields to the event loop between chunks — this is the biggest user-visible improvement
- **D-04:** Reduce `executeAsModal` scope to only wrap the actual `batchPlay` call, not the entire operation (base64 conversion, file I/O, etc.) — shortens the blocking window significantly

### React Render Optimization
- **D-05:** Targeted critical fixes only — no Draw.tsx structural refactor. Add `React.memo` to list components (HistoryList, HistoryItem, DownloadManager items), `useCallback` for expensive handlers (`applyInputValuesToPrompt`), `useMemo` for expensive derivations
- **D-06:** Do NOT split Draw.tsx into smaller components in this phase — too risky for a performance fix

### Scope Boundary
- **D-07:** This phase covers: 3 root causes (focus stealing, executeAsModal blocking, sync base64) PLUS easy secondary wins: console.log cleanup, temp file cleanup after export, fs.listDownloads optimization (use metadata instead of reading full content)
- **D-08:** Explicitly OUT OF SCOPE for this phase: duplicate message listener consolidation, Zustand store selector optimization, virtual scrolling for history list, Draw.tsx structural refactor, Web Worker offloading

### Claude's Discretion
- Exact whitelist of PS shortcuts to forward
- Chunk size for async base64 conversion
- Which console.log statements to remove vs keep for debugging
- Temp file cleanup strategy (immediate vs batch)

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `code/webapp/src/hooks/usePSBridge.ts` — Bridge communication hook, already has message infrastructure for forwarding commands to PS
- Bridge message protocol — UUID-based request-response pattern already established, can reuse for shortcut forwarding

### Established Patterns
- Bridge postMessage protocol — all PS-webapp communication uses structured messages with command + data
- Async operations in main.js — `readFile` returns promises, can be chained with async chunked base64
- Zustand stores — state management pattern, but components subscribe to entire store slices (no selectors)

### Integration Points
- Keydown handler must integrate with existing input field logic — only forward when NOT in an input/textarea
- Async base64 must work with existing Bridge response flow — results still need to be sent as base64 strings
- Temp file cleanup must not interfere with in-progress exports

<specifics>
## Specific Ideas

- The debug file has precise line numbers for every affected code location — planner should reference it directly
- The `String.fromCharCode.apply` pattern appears in 3 locations in main.js (lines 345-352, 515-520, 837-842) — all need the same async treatment
- `executeAsModal` wraps entire operation functions but only the `batchPlay` call needs modal state — the rest can run outside
- HistoryList/HistoryItem components re-render on any state change because they lack React.memo — high-impact, low-risk fix
- console.log cleanup is scattered but trivial — just remove them from production code

</specifics>
